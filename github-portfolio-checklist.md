# GitHub 作品集检查清单

在把项目 pin 到 GitHub 主页、写进简历或发给面试官之前，用这个清单检查。

## 仓库基础

- [ ] 仓库名清楚。
- [ ] README 开头先说明项目能做什么，而不是只列技术栈。
- [ ] 从干净 clone 开始，安装和启动步骤能跑通。
- [ ] `.env.example` 写清楚需要哪些环境变量。
- [ ] 没有提交 API key 或其他密钥。
- [ ] 有截图、demo GIF 或 demo 视频。
- [ ] 如果希望别人复用，添加 License。

## 工程质量

- [ ] 核心逻辑有测试。
- [ ] 错误状态有处理。
- [ ] LLM 调用、检索或 Agent 执行过程有日志。
- [ ] 依赖没有过度膨胀。
- [ ] 主要数据流在代码结构里容易看懂。

## AI 项目质量

- [ ] RAG 项目展示了文档导入、切块、embedding、检索和生成。
- [ ] Agent 项目展示了状态、工具调用和执行轨迹。
- [ ] README 说明失败案例和局限性。
- [ ] 有评估集或手动测试用例。
- [ ] 提到了成本、延迟和安全限制。

## 面试官视角

- [ ] 项目有 3-5 条清晰功能点。
- [ ] 项目有简单架构图。
- [ ] README 包含“学到了什么”和“下一步改进”。
- [ ] 简历 bullet 尽量有可衡量结果。
- [ ] 你能在 5 分钟内讲清楚这个项目。

## 四周发布门槛

下列条件全部满足后，才将当前版本标记为对外可复现里程碑：

- [ ] 从干净 clone 开始重跑安装、启动和核心流程。
- [ ] 全量测试或项目约定的验证命令退出码为 0。
- [ ] README 中的命令经过实际执行，不包含占位路径。
- [ ] 至少有一个截图、GIF 或短 demo。
- [ ] 至少记录一个真实失败案例及修复或降级方案。
- [ ] 架构图与当前代码一致。
- [ ] 已写一条可量化的简历 bullet 和一份 5 分钟讲解纲要。
- [ ] `trackers/project-scorecard.csv` 中的发布状态和证据路径已更新。
