# 前 12 周执行纪律

目标：不再盲目增加学习科目，而是提高完成质量。第一阶段的重点是先建立“能运行、能解释、能记录”的闭环，再逐步过渡到 Python 工程习惯、测试习惯和 GitHub 证据链。

路线权重：

- 70%：AI Agent / RAG / LLM 应用工程主线。
- 30%：Java + Spring Boot 后端实习兼容副线。

前 4 周不加入 Java。第 5 周开始 Java 只能作为低强度副线进入，每周必须绑定一个小产出，不能只看课。

## 每天最低要求

每天优先完成一个“最小闭环”，不要追求完美打卡：

- 一个主线动作：写一个小功能、修一个必改项、补一个测试，或读懂一个关键文件。
- 一个证据：运行结果、pytest 输出、截图、代码文件链接或 commit。
- 一份三行学习日志：今天完成、今天卡点、明天第一步。

证据必须对应真实文件或命令。不能把计划、未保存的代码或不存在的 commit 当成已完成证据。

状态好时再升级为完整闭环：

- 1 个小功能。
- 1-2 个运行证据；学到测试后再逐步加入 pytest。
- 1 篇三行学习日志；有重要发现时再补充可选证据。
- 1 次 commit。

AI 生成的代码在你能用自己的话解释关键逻辑、用测试或运行证据验证、并能完成一次小修改后，才可以计入当天产出。

## 每周硬性要求

每周必须完成：

- 1 个本周主交付物，其他学习内容都要服务它。
- 至少一次 Codex 作业批改。
- 至少一次根据批改意见进行修改。
- 至少 2 个运行证据。第 2 周开始再逐步要求 pytest。
- 至少一次质量提升：README、学习笔记、项目说明、测试或小重构，五选一即可。
- 周日晚复盘，并更新 `trackers/weekly-log.csv`。

每周交付物必须进入当前主项目。单独笔记、Fork、收藏、未迁移的教程练习和仅由 AI 生成但无法解释的代码，不计入周交付物。每 2 周验收一个技术里程碑，每 4 周验收一个可复现版本。

## 分阶段提升标准

### 第一次完成：入门练习

一个新知识点刚开始练时，先达到：

- 代码能运行。
- 你能解释每一段主要代码。
- 结果大体正确。
- 有学习日志和运行证据。

这个阶段不要求一次写成函数、测试、CLI 或 README。

### 第二次修改：小步改进

读懂第一版后，再做 1-2 个改进：

- 修明显 bug。
- 改清楚变量名。
- 减少重复。
- 可以尝试写一个简单函数。
- 补一个边界情况。

### 后续整理：工程化版本

当一个练习已经能解释清楚，再逐步升级为可复用模块。

## 每个练习的最终标准

每个 Python 练习最终可以达到：

- 有函数封装。
- 不依赖 `input()` 才能测试。
- 核心逻辑返回值清楚，优先返回 dict、list 或字符串。
- 可以被 pytest 调用。
- 能处理至少一个边界情况。
- 你能解释它对应的科班基础。

例如 `txt_check.py` 后续可以升级成：

- 文件名：`text_stats.py`。
- 核心函数：`count_text_stats(path)`。
- 返回结果：`{"lines": 3, "non_empty_lines": 2, "words": 10, "characters": 50}`。
- 测试文件：`tests/test_text_stats.py`。

## 批改和修改节奏

不要只追新任务。每天开始前先看昨天批改意见：

1. 有必改项：先修必改项。
2. 必改项修完：再做当天新功能。
3. 当天结束：提交作业批改。
4. 复查通过：再推送 GitHub。

如果连续两天没有完成必改项，第三天暂停新知识，只做修正。

## 降级和删减规则

计划服务于完成，不服务于制造负担。出现下面任意情况时，当天自动降级：

- 主线任务超过预计时间 30 分钟还没跑通。
- 连续两天没有完成必改项。
- 学习日志、算法、阅读和追踪表开始挤占写代码或改代码的时间。

降级后只保留：

- 修一个主线问题或必改项。
- 留一个运行证据或失败证据。
- 写三行学习日志。

降级当天可以删掉：

- 算法题 AC 要求，只保留读题、输入输出和伪代码。
- 长笔记和额外阅读。
- 自由探索格。
- 非必要的 README 美化和表格更新。

## Java 副线进入规则

### 第 1-4 周：不加 Java

先把 Python 函数化、pytest、CLI、GitHub 习惯跑通。这个阶段不做 Java / Spring Boot，避免同时开太多战线。

### 第 5-8 周：每周 2 小时 Java 基础

允许学习：

- Java 基础语法。
- 集合、类、异常。
- Maven 项目结构。
- JUnit 最小测试。

每周至少交付一个小产出：

- 一个 Java 小函数；
- 一个集合练习；
- 一个 JUnit 测试；
- 或一个简短 README，说明 Java 练习和后端能力的关系。

### 第 9-12 周：每周 3-4 小时 Java 后端

允许学习：

- Spring Boot REST API。
- JSON 请求和响应。
- 参数校验。
- 基础 CRUD。
- PostgreSQL 或 MySQL 连接。

每周至少交付一个后端产出：

- 一个可运行 API；
- 一个 Controller / Service / Repository 小闭环；
- 一个数据库表设计；
- 一个接口测试；
- 或一次 Java 后端 README 更新。

## 暂时不增加的内容

前 12 周先不要主动扩展这些内容：

- 数据库系统深入。
- LangGraph。
- RAG 完整系统。
- Docker 深入。
- 大量算法刷题。
- Spring Cloud。
- JVM 调优。
- 高并发深入。
- Kubernetes。

它们不是不重要，而是还没到当前阶段。现在先把 Python 工程、测试、CLI、Next.js、API、LLM 接入，以及 Java 后端最小闭环做扎实。算法只保留每日 20-30 分钟微练习，用来巩固字符串、数组、哈希表、栈、队列等基础数据结构，不升级成题海。

为了避免计划把学习变成纯打卡，每天允许一个可选自由探索格：20-40 分钟，只追一个好奇问题。自由探索可以不产出代码，但要留下一句话问题或发现；它不能替代主线交付，也不需要补债。

## 每周自检问题

周日复盘时回答：

1. 本周哪个代码文件从“脚本”升级成了“可测试模块”？
2. 本周至少 2 个测试在哪里？
3. 本周哪条批改意见已经修完？
4. 本周 GitHub 上有什么证据？
5. 本周 Java 副线是否有可见产出？第 1-4 周填“不适用”。
6. 下周要减少什么无效学习？
