# 前 12 周执行纪律

目标：不再盲目增加学习科目，也不被预写日程推动。第一阶段先建立“自己提出问题、用实验回答、能运行、能解释、能记录”的闭环，再逐步过渡到工程习惯和 GitHub 证据链。

路线权重：

- 70%：AI Agent / RAG / LLM 应用工程主线。
- 30%：Java + Spring Boot 后端实习兼容副线。

前 4 周不加入 Java。第 5 周开始 Java 只能作为低强度副线进入，每周必须绑定一个小产出，不能只看课。

## 单次学习最低要求

每次学习优先完成一个“最小实验”，不要追求完美打卡：

- 一个主线动作：验证一个假设、写一个小功能、修一个关键问题、补一个测试，或读懂一个关键文件。
- 一个证据：运行结果、pytest 输出、截图、代码文件链接或 commit。
- 一份三行学习日志：验证了什么、当前证据、下一次最自然的动作。

证据必须对应真实文件或命令。不能把计划、未保存的代码或不存在的 commit 当成已完成证据。

状态好时再升级为完整闭环：

- 1 个小功能。
- 1-2 个运行证据；学到测试后再逐步加入 pytest。
- 1 篇三行学习日志；有重要发现时再补充可选证据。
- 1 次 commit。

AI 生成的代码在你能用自己的话解释关键逻辑、用测试或运行证据验证、并能完成一次小修改后，才可以计入当天产出。

## 每周硬性要求

每周必须完成：

- 1 个本周主交付物，其他学习内容都要服务它。
- 一次周中无评分诊断。
- 一次周末项目验收，并记录继续、转向或暂存。
- 至少 2 个运行证据。第 2 周开始再逐步要求 pytest。
- 至少一次质量提升：README、学习笔记、项目说明、测试或小重构，五选一即可。
- 周末复盘，并更新 `trackers/weekly-log.csv`。
- 实际使用 4 小时相邻自由探索；可以没有代码产出，但要留下问题和发现。

每周交付物必须进入当前主项目。单独笔记、Fork、收藏、未迁移的教程练习和仅由 AI 生成但无法解释的代码，不计入周交付物。每 2 周验收一个技术里程碑，每 4 周验收一个可复现版本。

## 分阶段提升标准

### 第一次完成：入门练习

一个新知识点刚开始练时，先达到：

- 代码能运行。
- 你能解释每一段主要代码。
- 结果大体正确。
- 有学习日志和运行证据。

这个阶段不要求一次写成函数、测试、CLI 或 README。

### 第二次修改：小步改进

读懂第一版后，再做 1-2 个改进：

- 修明显 bug。
- 改清楚变量名。
- 减少重复。
- 可以尝试写一个简单函数。
- 补一个边界情况。

### 后续整理：工程化版本

当一个练习已经能解释清楚，再逐步升级为可复用模块。

## 每个练习的最终标准

每个 Python 练习最终可以达到：

- 有函数封装。
- 不依赖 `input()` 才能测试。
- 核心逻辑返回值清楚，优先返回 dict、list 或字符串。
- 可以被 pytest 调用。
- 能处理至少一个边界情况。
- 你能解释它对应的科班基础。

例如 `txt_check.py` 后续可以升级成：

- 文件名：`text_stats.py`。
- 核心函数：`count_text_stats(path)`。
- 返回结果：`{"lines": 3, "non_empty_lines": 2, "words": 10, "characters": 50}`。
- 测试文件：`tests/test_text_stats.py`。

## 诊断和修改节奏

不要为了完成任务卡而追新知识。每次开始前先看本周主问题和当前最大未知点：

1. 能用小实验验证：直接实验。
2. 缺少必要知识：从任务库选择一张对应卡片。
3. 周中仍卡住：提交诊断，调整任务池或缩小交付物。
4. 周末按项目证据验收，不按完成任务卡数量评分。

关键正确性问题仍需修复，但它属于项目风险，不形成跨日作业债务。

## 降级和删减规则

计划服务于完成，不服务于制造负担。出现下面任意情况时，本次学习自动降级：

- 主线任务超过预计时间 30 分钟还没跑通。
- 连续两次实验没有减少不确定性。
- 学习日志、额外阅读和追踪表开始挤占写代码或验证的时间。

降级后只保留：

- 验证一个主线假设或修一个关键问题。
- 留一个运行证据或失败证据。
- 写三行学习日志。

降级时可以删掉：

- 算法题 AC 要求，只保留读题、输入输出和伪代码。
- 长笔记和额外阅读。
- 非必要的 README 美化和表格更新。

受保护的 4 小时自由探索不能删除，只能移动到本周其他时段。

## Java 副线进入规则

### 第 1-4 周：不加 Java

先把 Python 函数化、pytest、CLI、GitHub 习惯跑通。这个阶段不做 Java / Spring Boot，避免同时开太多战线。

### 第 5-8 周：每周 2 小时 Java 基础

允许学习：

- Java 基础语法。
- 集合、类、异常。
- Maven 项目结构。
- JUnit 最小测试。

每周至少交付一个小产出：

- 一个 Java 小函数；
- 一个集合练习；
- 一个 JUnit 测试；
- 或一个简短 README，说明 Java 练习和后端能力的关系。

### 第 9-12 周：每周 3-4 小时 Java 后端

允许学习：

- Spring Boot REST API。
- JSON 请求和响应。
- 参数校验。
- 基础 CRUD。
- PostgreSQL 或 MySQL 连接。

每周至少交付一个后端产出：

- 一个可运行 API；
- 一个 Controller / Service / Repository 小闭环；
- 一个数据库表设计；
- 一个接口测试；
- 或一次 Java 后端 README 更新。

## 暂时不增加的内容

前 12 周先不要主动扩展这些内容：

- 数据库系统深入。
- LangGraph。
- RAG 完整系统。
- Docker 深入。
- 大量算法刷题。
- Spring Cloud。
- JVM 调优。
- 高并发深入。
- Kubernetes。

它们不是不重要，而是还没到当前阶段。现在先把 Python 工程、测试、CLI、Next.js、API、LLM 接入，以及 Java 后端最小闭环做扎实。算法只在项目卡点或独立面试准备需要时进入，不升级成每日题海。

为了避免计划把学习变成纯打卡，每周固定保留 4 小时相邻自由探索。它不是完成主线后的奖励，不因主线延期取消，也不要求当周转化为代码。只有当探索被提升为正式方向时，才进入项目资产链。

第 5-12 周预写任务不再是日程。任务文件保留为按卡点调用的知识和实验卡；未选择、未完成或跳过任务卡都不形成补债。

## 每周自检问题

周日复盘时回答：

1. 本周主问题是什么，最终回答是什么？
2. 哪两个证据支持这个回答？
3. 本周最大的未知点是否被实验缩小？
4. 本周 GitHub 或本地项目新增了什么真实能力？
5. 4 小时相邻探索实际用了多少，发现了什么？
6. 下周应该继续、转向还是暂存？
