
`tmp_path` 本质上就是 **pytest 帮你创建并管理的临时目录**。

如果从第一性原理理解：

---

## 不用 tmp_path 会怎样？

假设你要测试一个生成 JSON 文件的函数：

```python
def create_json(data, output_path):
    ...
```

测试时你可能会这样写：

```python
def test_create_json():
    create_json({"a": 1}, "output/test.json")
```

问题来了：

- 会污染项目目录

- 测试之间互相影响

- 运行多次会留下垃圾文件

- CI环境可能没有这个目录


所以测试需要一个：

> 「随便写、随便删、每个测试独立」的目录

这就是 `tmp_path`。

---

## tmp_path 是什么？

pytest 自动注入的 fixture：

```python
def test_example(tmp_path):
    pass
```

这里的：

```python
tmp_path
```

实际上是一个：

```python
pathlib.Path
```

对象。

你可以打印：

```python
def test_example(tmp_path):
    print(tmp_path)
```

例如：

```python
PosixPath('/tmp/pytest-of-kris/pytest-0/test_example0')
```

pytest 会提前创建好这个目录。

---

## 可以把它当成一个空文件夹

例如：

```python
def test_example(tmp_path):
    file = tmp_path / "hello.txt"

    file.write_text("hello")

    assert file.read_text() == "hello"
```

运行前：

```text
/tmp/pytest-xxx/test_example0/
```

运行后：

```text
/tmp/pytest-xxx/test_example0/
└── hello.txt
```

整个过程不会污染你的项目目录。

---

## 和你现在的 create_json 结合

你的代码：

```python
def create_json(json_result, output_path):
    ...
```

测试时：

```python
def test_create_json(tmp_path):
    output_file = tmp_path / "chunks.json"

    create_json(
        {"name": "kris"},
        output_file
    )

    assert output_file.exists()
```

这里：

```python
tmp_path
```

相当于：

```text
/tmp/pytest-xxx/test_create_json0/
```

而：

```python
output_file = tmp_path / "chunks.json"
```

变成：

```text
/tmp/pytest-xxx/test_create_json0/chunks.json
```

你的 JSON 文件只会写到临时目录。

---

## 为什么 Path 对象特别方便

因为：

```python
tmp_path
```

本身就是：

```python
Path
```

所以可以直接：

### 拼接路径

```python
tmp_path / "output" / "chunks.json"
```

得到：

```text
/tmp/pytest-xxx/output/chunks.json
```

---

### 创建目录

```python
(tmp_path / "output").mkdir()
```

---

### 判断文件是否存在

```python
assert output_file.exists()
```

---

### 读取内容

```python
content = output_file.read_text()
```

---

## 典型测试模式

对于你现在学的工程化 Python，80% 的文件测试都是这个套路：

```python
def test_xxx(tmp_path):
    # 1. 准备路径
    output_file = tmp_path / "result.json"

    # 2. 调用函数
    create_json(data, output_file)

    # 3. 验证文件存在
    assert output_file.exists()

    # 4. 验证文件内容
    content = json.loads(
        output_file.read_text(encoding="utf-8")
    )

    assert content == data
```

这就是标准的：

```text
Arrange
↓
Act
↓
Assert
```

测试结构。

## tmp_path 和 tmp_path_factory 的区别

### tmp_path

每个测试一个目录

```python
def test_a(tmp_path):
    ...

def test_b(tmp_path):
    ...
```

得到：

```text
test_a0/
test_b0/
```

互不影响。

---

### tmp_path_factory

整个测试会话共享

```python
@pytest.fixture(scope="session")
def data_dir(tmp_path_factory):
    return tmp_path_factory.mktemp("data")
```

只创建一次：

```text
data/
```

适合：

- 大模型文件

- 图片

- 数据集

- 向量库


这种生成一次很贵的资源。

---

## 一句话记忆

可以把 `tmp_path` 理解成：

> pytest 给每个测试自动分配的、独立的、会自动清理的 `Path` 对象临时工作区。

以后你写：

- JSON输出测试

- CSV输出测试

- 日志文件测试

- 图片生成测试

- 文本切块工具测试


基本都会用到 `tmp_path`。对于你现在在做的 `create_json()` 项目，它几乎是最常用的 fixture 之一。
